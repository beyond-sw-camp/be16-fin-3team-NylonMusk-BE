# 🚀 MKX Delisting Auto System - 완전 자동화 테스트 가이드

## 📋 개요

완전 자동화된 상장폐지 위험 감지 시스템의 모든 기능을 테스트하는 가이드입니다.

## 💰 새로운 3단계 보상금 지급 시스템

### 🎯 지급 방식

1. **1단계: 기업 계좌 현금 지급**

   - 기업 계좌에 충분한 잔액이 있으면 정상 지급
   - 모든 보상금을 현금으로 완전 지급

2. **2단계: 유동자산 활용 지급**

   - 기업 계좌 잔액 부족 시 유동자산 활용
   - 유동자산의 70%까지 활용 (30%는 운영자금으로 보존)
   - 현금 + 유동자산으로 충분하면 완전 지급
   - 부족하면 비례 부분 지급 (PARTIAL_PAID 상태)

3. **3단계: 파산 처리 후 거래소 지급**
   - 유동자산으로도 부족하면 기업 파산 처리
   - 거래소 운영 계좌에서 미지급분 지급
   - **거래소 지원금을 대출로 기록** (ExchangeSupportFund)
   - 거래소 계좌도 부족하면 최종 실패

### 📊 보상금 상태

- **PENDING**: 보상금 생성됨
- **PROCESSING**: 보상금 처리 중
- **COMPLETED**: 보상금 지급 완료
- **PARTIAL_PAID**: 보상금 부분 지급 (새로 추가)
- **FAILED**: 보상금 지급 실패
- **CANCELLED**: 보상금 취소됨

### 🔄 자동 재처리

- 실패한 보상금은 자동으로 재처리 시도
- 스케줄러가 10분마다 실패한 보상금 재처리
- 관리자가 수동으로 재처리 가능

## 🏦 거래소 지원금 관리 시스템

### 지원금 기록 (ExchangeSupportFund)

거래소에서 지급한 지원금은 **대출**로 기록되어 관리됩니다.

#### 지원금 유형

- **COMPENSATION_LOAN**: 상장폐지 보상금 대출
- **OPERATING_LOAN**: 기업 운영자금 대출
- **EMERGENCY_LOAN**: 긴급자금 대출
- **GUARANTEE**: 보증 제공
- **DIRECT_SUPPORT**: 직접 지원금

#### 지원금 상태

- **ACTIVE**: 활성 대출
- **REPAID**: 상환완료
- **PARTIAL_REPAID**: 부분상환
- **OVERDUE**: 연체
- **WRITTEN_OFF**: 대손처리
- **CANCELLED**: 취소

### 지원금 관리 기능

#### 1. 상환 처리

- 부분 상환 및 완전 상환 지원
- 잔여 금액 자동 계산
- 상태 자동 업데이트

#### 2. 연체 관리

- 상환 예정일 기반 연체 자동 감지
- 연체 상태 자동 변경

#### 3. 대손 처리

- 회수 불가능한 대출 대손 처리
- 손실 인정 및 정리

## 🎯 테스트 시나리오

### 📊 섹션 1: 기준 관리

**목적**: 상장폐지 기준 생성 및 관리

#### 🚀 자동 생성된 기본 기준들

애플리케이션 시작 시 다음 기준들이 **자동으로 생성**됩니다:

##### 재무 기준 (10개):

- **LOW_REVENUE_2Y**: 2년간 평균 매출액 50억원 미만
- **NEGATIVE_NET_INCOME**: 순이익 마이너스
- **LOW_EQUITY_RATIO**: 자기자본비율 20% 미만
- **HIGH_DEBT_RATIO**: 부채비율 200% 초과
- **LOW_CURRENT_RATIO**: 유동비율 1.0 미만
- **INSUFFICIENT_CAPITAL**: 자본금 10억원 미만
- **LOW_ROE**: ROE 5% 미만
- **LOW_ROA**: ROA 3% 미만
- **NEGATIVE_OPERATING_INCOME**: 영업이익 마이너스
- **LOW_INTEREST_COVERAGE**: 이자보상배수 1.0 미만

##### 거래 기준 (4개):

- **LOW_TRADING_VOLUME_2Q**: 거래량 부족 2분기 연속
- **LOW_STOCK_PRICE_30D**: 30일 평균 주가 1000원 미만
- **LOW_MARKET_CAP_CONTINUOUS**: 저시가총액 지속
- **LOW_MINORITY_SHAREHOLDERS**: 소액주주 200인 미만 2년 연속

##### 법규 기준 (7개):

- **AUDIT_OPINION_ADVERSE**: 감사 의견 부적정
- **AUDIT_OPINION_QUALIFIED**: 감사 의견 한정
- **AUDIT_OPINION_DISCLAIMER**: 감사 의견 거절
- **REPORT_DELAY**: 보고서 제출 지연 10일 초과
- **FINAL_BANKRUPTCY**: 최종 부도
- **EMBEZZLEMENT**: 횡령 발생
- **BREACH_OF_TRUST**: 배임 발생

### 🔍 섹션 2: 위반 감지 및 관리

**목적**: 상장폐지 기준 위반 감지 및 관리

#### 테스트 순서:

1. **2.1 수동 위반 감지**: 특정 주식의 위반 상황 수동 등록
2. **2.2 위반 목록 조회**: 등록된 위반 목록 확인
3. **2.3 위반 해결**: 위반 상황 해결 처리

### 🚨 섹션 3: 상장폐지 실행

**목적**: 전체 상장폐지 프로세스 테스트

#### 테스트 순서:

1. **3.1 상장폐지 예고**: 주주들에게 상장폐지 예고 공지
2. **3.2 상장폐지 실행**: 실제 상장폐지 처리 및 보상금 지급
3. **3.3 보상금 재처리**: 실패한 보상금 재처리

### 📈 섹션 4: 자동 감지 시스템

**목적**: 재무제표 기반 자동 위반 감지 테스트

#### 테스트 순서:

1. **4.1 재무제표 기반 자동 감지**: 실제 재무 데이터로 자동 감지
2. **4.2 상장폐지 위험 분석 리포트**: 종합적인 위험도 분석

### 🔄 섹션 5: 에러 케이스 테스트

**목적**: 예외 상황 및 에러 처리 테스트

#### 테스트 케이스:

1. **5.1 존재하지 않는 주식 ID**: 잘못된 주식 ID로 요청
2. **5.2 재무제표 데이터 없음**: 재무 데이터가 없는 주식 테스트

### 🏦 섹션 6: 거래소 지원금 관리

**목적**: 거래소 지원금 관리 시스템 테스트

#### 테스트 순서:

1. **6.1 모든 지원금 조회**: 거래소 전체 지원금 현황
2. **6.2 주식별 지원금 조회**: 특정 주식의 지원금 기록
3. **6.3 기업별 지원금 조회**: 특정 기업의 지원금 현황
4. **6.4 연체된 지원금 조회**: 연체된 지원금 목록
5. **6.5 거래소 총 지원금 합계**: 거래소 전체 지원금 통계
6. **6.6 거래소 총 미상환 금액**: 회수해야 할 총 금액
7. **6.7 지원금 상환 처리**: 부분/완전 상환 처리
8. **6.8 지원금 연체 처리**: 연체 상태 변경
9. **6.9 지원금 대손 처리**: 대손 처리

## 📊 실제 처리 시나리오

### 시나리오 1: 정상 지급

```
총 보상금: 10억원
기업 계좌 잔액: 15억원
→ 1단계: 현금으로 완전 지급 ✅
```

### 시나리오 2: 유동자산 활용

```
총 보상금: 10억원
기업 계좌 잔액: 3억원
유동자산: 12억원 (사용가능: 8.4억원)
→ 2단계: 현금 3억 + 유동자산 7억 = 완전 지급 ✅
```

### 시나리오 3: 부분 지급 + 거래소 지원

```
총 보상금: 10억원
기업 계좌 잔액: 2억원
유동자산: 8억원 (사용가능: 5.6억원)
→ 2단계: 현금 2억 + 유동자산 5.6억 = 7.6억원 부분 지급
→ 3단계: 거래소에서 2.4억원 추가 지급
→ 거래소 지원금 기록: 2.4억원 (3년 후 상환, 연 5% 이자) ✅
```

### 시나리오 4: 최종 실패

```
총 보상금: 10억원
기업 계좌 잔액: 1억원
유동자산: 2억원 (사용가능: 1.4억원)
거래소 운영 계좌: 1억원
→ 1+2+3단계 모두 부족 → 최종 실패 ❌
```

## 🛡️ 안전장치 및 예외처리

### 1. 타입 안전성

- `BigDecimal` 사용으로 정밀한 금액 계산
- `BigInteger`와 `BigDecimal` 간 적절한 변환

### 2. 트랜잭션 관리

- `@Transactional`로 데이터 일관성 보장
- 실패 시 롤백 처리

### 3. 예외 처리

- FeignClient 호출 실패 시 대체 로직
- 계좌 잔액 부족 시 적절한 예외 발생
- 상세한 로그 기록으로 추적 가능

### 4. 멱등성 보장

- 재처리 시 중복 지급 방지
- 상태 기반 처리로 안전성 확보

## 📊 모니터링 및 로깅

### 로그 레벨

- `INFO`: 정상 처리 과정
- `WARN`: 부분 지급, 자산 부족
- `ERROR`: 처리 실패, 예외 상황

### 이력 관리

- `DelistingHistory`로 모든 처리 과정 기록
- `ActionType.COMPENSATION_FAILED`로 실패 추적
- 재처리 시도 기록으로 관리 가능

## 🚀 실제 금융 환경 적용 가능성

### 장점

1. **현실적인 처리 방식**: 실제 증권거래소와 유사한 3단계 구조
2. **운영자금 보존**: 유동자산의 30% 보존으로 기업 운영 보장
3. **자동화**: 실패 시 자동 재처리로 관리 부담 최소화
4. **투명성**: 모든 처리 과정의 상세한 이력 관리
5. **확장성**: MSA 구조로 각 서비스 독립적 확장 가능
6. **거래소 자산 관리**: 지원금을 대출로 기록하여 회수 관리

### 개선 가능 영역

1. **자산 매각 시스템**: 부동산, 장비 등 비유동자산 매각 로직
2. **보험 처리**: 상장폐지 보험 등 추가 보상 수단
3. **법적 절차**: 파산 신청, 법원 승인 등 법적 프로세스
4. **국제 표준**: IFRS, K-IFRS 등 국제 회계 기준 적용
